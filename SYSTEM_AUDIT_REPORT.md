**技术栈**: Next.js 16.1.1 + React 19.2.3 + FastAPI + PostgreSQL + Redis + WebSocket

## 一、前端 UX 界面问题

### 1.1 违反编码规范问题

#### 问题 1.1.1: 使用 console.log/console.error 而非结构化日志
- **具体现象**: 多个前端组件直接使用 `console.log`、`console.error` 输出调试信息
- **涉及文件**:
  - `web/src/components/ErrorBoundary.tsx` (行 60, 205)
  - `web/src/components/highlights/HighlightCard.tsx` (行 86, 101)
  - `web/src/components/highlights/HighlightDetailModal.tsx` (行 140, 162)
  - `web/src/components/training/ScenarioList.tsx` (行 38)
  - `web/src/components/ui/audio-visualizer.tsx` (行 104)
  - `web/src/components/admin/knowledge-answer/tabs/intent-rules-tab.tsx` (行 55)
  - `web/src/app/test-mic/page.tsx` (行 18)
  - `web/src/app/admin/settings/page.tsx` (行 258, 305, 348, 371, 415)
- **可能原因**: 开发过程中遗留的调试代码未清理
- **影响范围**: 生产环境可能泄露敏感信息，影响性能
- **涉及环节**: 前端
- **修复建议**: 
  - 统一使用 `@/lib/debug` 模块的 debug.log/debug.error
  - 生产环境自动禁用调试输出
- **优先级**: 中

#### 问题 1.1.2: 使用 alert() 和 confirm() 弹窗
- **具体现象**: 管理后台页面使用原生弹窗，违反"用户体验永不中断"原则
- **涉及文件**:
  - `web/src/app/admin/rag-profiles/page.tsx` (行 184: `confirm("确认删除该配置？")`)
  - `web/src/app/admin/records/page.tsx` (行 68: `confirm("确定要删除这条训练记录吗？")`, 行 75: `alert("删除失败")`)
  - `web/src/app/admin/personas/[id]/page.tsx` (行 439: `alert("请输入角色名称")`, 行 443: `alert("请输入系统提示词")`, 行 477: `alert("保存失败...")`, 行 1132: `alert("音频播放失败")`, 行 1138: `alert("试听失败")`)
- **可能原因**: 快速开发时使用原生弹窗，未遵循设计规范
- **影响范围**: 用户体验中断，不符合企业级应用标准
- **涉及环节**: 前端 UI
- **修复建议**: 
  - 使用 Radix UI Dialog 组件替代 confirm/alert
  - 删除操作使用二次确认对话框
  - 错误提示使用 Toast 或状态指示器
- **优先级**: 高

#### 问题 1.1.3: 使用 window.location 直接跳转
- **具体现象**: 部分页面使用 `window.location` 而非 Next.js router
- **涉及文件**:
  - `web/src/components/ErrorBoundary.tsx` (行 80, 98)
  - `web/src/components/layout/admin-shell.tsx` (行 34, 39)
  - `web/src/components/layout/dashboard-shell.tsx` (行 31)
  - `web/src/app/admin/error.tsx` (行 35)
  - `web/src/app/(dashboard)/profile/page.tsx` (行 349)
- **可能原因**: 未遵循 Next.js 最佳实践
- **影响范围**: 页面跳转不经过 Next.js 路由系统，可能丢失状态
- **涉及环节**: 前端路由
- **修复建议**: 
  - 统一使用 `useRouter` hook 的 `router.push()` / `router.replace()`
  - ErrorBoundary 可使用 window.location.reload()（这是合理用法）
- **优先级**: 中

#### 问题 1.1.4: 使用 bg-white 而非 bg-stone-50
- **具体现象**: 部分组件使用纯白背景，违反设计规范
- **涉及文件**:
  - `web/src/components/ErrorBoundary.tsx` (行 111)
  - `web/src/components/layout/admin-shell.tsx` (行 86)
  - `web/src/components/layout/dashboard-shell.tsx` (行 78)
  - `web/src/components/layout/sidebar.tsx` (行 239)
  - `web/src/components/layout/admin-sidebar.tsx` (行 274, 292)
  - `web/src/components/practice/ScorePanel.tsx` (行 179, 190)
  - `web/src/components/practice/presentation/PageNavigator.tsx` (行 34)
  - `web/src/components/practice/presentation/SlideViewer.tsx` (行 96, 105)
  - `web/src/components/practice/presentation/PointTracker.tsx` (行 24)
- **可能原因**: 未统一设计规范，或临时修改未同步
- **影响范围**: 视觉不一致，影响品牌形象
- **涉及环节**: 前端样式
- **修复建议**: 
  - 统一使用 `bg-stone-50` 或 `bg-white/60` 等半透明背景
  - 建立设计 Token 统一管理
- **优先级**: 低

### 1.2 响应式适配问题

#### 问题 1.2.1: 移动端布局可能存在溢出
- **具体现象**: 部分页面在移动端可能存在横向滚动
- **涉及文件**:
  - `web/src/app/(dashboard)/page.tsx` - Dashboard 页面在移动端可能需要优化间距
  - `web/src/app/(user)/practice/[sessionId]/page.tsx` - 练习页面在移动端按钮尺寸可能过大
  - `web/src/app/admin/page.tsx` - 管理后台在移动端可能布局拥挤
- **可能原因**: 响应式断点设置不够细致
- **影响范围**: 移动端用户体验
- **涉及环节**: 前端响应式设计
- **修复建议**: 
  - 增加移动端专用样式测试
  - 使用 Chrome DevTools 设备模式验证
  - 考虑为移动端简化布局
- **优先级**: 中

### 1.3 交互状态问题

#### 问题 1.3.1: 录音按钮防抖逻辑可能不足
- **具体现象**: `web/src/app/(user)/practice/[sessionId]/page.tsx` 中录音切换逻辑使用 300ms 防抖，但在网络延迟情况下可能仍导致重复触发
- **涉及文件**: `web/src/app/(user)/practice/[sessionId]/page.tsx` (行 286-304)
- **可能原因**: 防抖时间设置过短，未考虑网络延迟
- **影响范围**: 可能导致重复录音开始，影响用户体验
- **涉及环节**: 前端交互逻辑
- **修复建议**: 
  - 增加防抖时间至 500ms
  - 添加更严格的状态检查
  - 考虑使用禁用状态而非防抖
- **优先级**: 中

## 二、后端 API 与业务逻辑问题

### 2.1 违反编码规范问题

#### 问题 2.1.1: 使用 print() 而非 logger
- **具体现象**: 后端代码中存在 print() 语句
- **涉及文件**:
  - `backend/src/evaluation/schemas.py` (行 171)
  - `backend/src/agent/capabilities/registry.py` (行 200, 252)
  - `backend/src/common/ai/encryption.py` (行 31)
- **可能原因**: 临时调试代码未清理
- **影响范围**: 生产环境日志不规范，难以追踪
- **涉及环节**: 后端日志
- **修复建议**: 
  - 统一使用 `get_logger(__name__).info()` / `debug()`
  - 移除所有 print() 语句
- **优先级**: 中

#### 问题 2.1.2: 使用 HTTPException 而非 Result[T] 模式
- **具体现象**: 大量 API 路由直接抛出 HTTPException，违反错误处理规范
- **涉及文件**:
  - `backend/src/prompt_templates/api/routes.py` (13 处 HTTPException)
  - `backend/src/main.py` (行 313)
  - `backend/src/presentation_coach/api/presentations.py` (8 处 HTTPException)
- **可能原因**: 开发时直接使用 FastAPI 默认异常处理
- **影响范围**: 错误处理不统一，违反"用户体验永不中断"原则
- **涉及环节**: 后端 API 错误处理
- **修复建议**: 
  - 统一使用 `Result.fail()` 包装错误
  - 通过中间件统一转换为用户友好的响应
- **优先级**: 高

#### 问题 2.1.3: TODO 标记未处理
- **具体现象**: 代码中存在 TODO 标记但未处理
- **涉及文件**:
  - `backend/src/evaluation/services/staged_evaluation.py` (行 112: `# TODO: parameterize`)
- **可能原因**: 功能未完成或临时方案
- **影响范围**: 代码可维护性，可能影响功能完整性
- **涉及环节**: 后端业务逻辑
- **修复建议**: 
  - 完成 TODO 标记的功能
  - 或转换为 Issue 追踪
- **优先级**: 低

### 2.2 异常处理问题

#### 问题 2.2.1: 异常处理不够细致
- **具体现象**: 部分异常处理使用通用的 `except Exception`，可能掩盖具体错误
- **涉及文件**:
  - `backend/src/prompt_templates/models.py` (行 136: `except Exception:`)
  - `backend/src/presentation_coach/services/interruption_detector.py` (行 40, 124)
- **可能原因**: 快速开发时使用通用异常捕获
- **影响范围**: 难以定位具体错误，影响调试效率
- **涉及环节**: 后端异常处理
- **修复建议**: 
  - 捕获具体的异常类型
  - 记录详细的错误堆栈
- **优先级**: 中

### 2.3 业务逻辑问题

#### 问题 2.3.1: 会话状态流转可能存在竞态条件
- **具体现象**: SessionLifecycleService 的状态转换逻辑在并发场景下可能存在问题
- **涉及文件**: `backend/src/common/db/session_lifecycle.py`
- **可能原因**: 未使用数据库锁或乐观并发控制
- **影响范围**: 并发操作时可能导致状态不一致
- **涉及环节**: 后端状态管理
- **修复建议**: 
  - 添加数据库行锁
  - 或使用版本号实现乐观并发控制
- **优先级**: 高

## 三、前后端联调问题

### 3.1 接口调用问题

#### 问题 3.1.1: API 错误处理不统一
- **具体现象**: 前端 API 客户端使用统一的错误处理，但部分后端接口返回格式不一致
- **涉及文件**:
  - `web/src/lib/api/client.ts` - API 客户端定义
  - 后端多个 API 路由
- **可能原因**: 后端未严格遵循统一的错误响应格式
- **影响范围**: 前端错误处理可能失效
- **涉及环节**: 前后端联调
- **修复建议**: 
  - 统一所有 API 的错误响应格式
  - 添加 API 契约测试
- **优先级**: 高

### 3.2 WebSocket 通信问题

#### 问题 3.2.1: WebSocket 连接管理复杂度高
- **具体现象**: `usePracticeWebSocket` hook 文件过大（844 行），逻辑复杂
- **涉及文件**: 
  - `web/src/hooks/use-practice-websocket.ts`
  - `backend/src/sales_bot/websocket/stepfun_realtime_handler.py` (4511 行)
- **可能原因**: 功能累积导致单个文件过于复杂
- **影响范围**: 维护困难，容易引入 bug
- **涉及环节**: WebSocket 实时通信
- **修复建议**: 
  - 进一步拆分模块（虽然已经拆分过一次）
  - 考虑使用状态机模式管理连接状态
- **优先级**: 中

#### 问题 3.2.2: WebSocket 重连逻辑可能不够健壮

- **可能原因**: 简化实现
- **影响范围**: 网络不稳定时重连策略不够智能
- **涉及环节**: WebSocket 连接管理
- **修复建议**: 
  - 实现指数退避重连策略
  - 增加网络状态检测
- **优先级**: 中

## 四、权限控制与安全问题

### 4.1 权限控制问题

#### 问题 4.1.1: JWT Secret 默认值不安全
- **具体现象**: JWT_SECRET 默认值为 "your-super-secret-key-change-in-production-min-32-chars"
- **涉及文件**: `backend/src/common/auth/service.py` (行 28)
- **可能原因**: 开发环境默认值
- **影响范围**: 生产环境如果未修改会导致严重安全风险
- **涉及环节**: 后端安全
- **修复建议**: 
  - 已在 main.py 中添加启动时检查（行 161-165）
  - 确保生产环境强制配置
- **优先级**: 高（已部分缓解）

#### 问题 4.1.2: 权限检查不够细粒度
- **具体现象**: 部分管理后台 API 可能缺少细粒度的权限检查
- **涉及文件**: 
  - `backend/src/admin/api/` 下的多个路由
- **可能原因**: 使用粗粒度的角色检查
- **影响范围**: 可能存在越权访问风险
- **涉及环节**: 后端权限控制
- **修复建议**: 
  - 实现基于资源的权限检查（RBAC）
  - 添加权限审计日志
- **优先级**: 高

### 4.2 数据隐私问题

#### 问题 4.2.1: 敏感信息可能通过日志泄露
- **具体现象**: 日志中可能包含用户 token、个人信息等敏感数据
- **涉及文件**: 多个日志输出位置
- **可能原因**: 未对日志进行脱敏处理
- **影响范围**: 数据隐私合规风险
- **涉及环节**: 后端日志安全
- **修复建议**: 
  - 实现日志脱敏中间件
  - 对 token、密码等敏感字段进行掩码
- **优先级**: 高
