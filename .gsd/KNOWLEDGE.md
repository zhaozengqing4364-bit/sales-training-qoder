- M022/S01 methodology gotcha: current sales runtime still has no standalone `qualification` stage ŌĆö `sales_stage` only ships `opening / discovery / presentation / objection / closing`. If a later slice wires methodology-aware rubric logic, qualification evidence must stay merged into `opening + discovery` until the stage contract itself changes; otherwise realtime/report/history/admin will silently invent different interpretations of the same session.
- M022/S01 methodology consumption seam: sales methodology status now ships additively on `canonical_evaluation_kernel.methodology` and is mirrored for transition consumers through `compatibility_readers.sales_methodology_rubric_v1`. If a later slice needs learner/report/history/admin/manager methodology semantics, read one of those shared payloads instead of inventing new top-level fields or re-deriving rubric health from local score aliases.
- M022/S03 manager/admin truth-surface guardrail: do not create a second homepage summary line for supervisors. The current productized truth surfaces are `web/src/components/admin/manager-lite-panel.tsx`, `web/src/app/admin/analytics/page.tsx`, and `web/src/app/admin/users/[id]/page.tsx`/`backend/src/admin/api/users.py`; admin-home cards without a real backend authority must stay downgraded to inventory/link copy instead of demo metrics or manager-only local rollups.
- M022/S04 org-boundary gotcha: the shipped system has no org-aware owner columns yet ŌĆö most runtime truth still hangs off `practice_sessions.user_id`, global `users.role`, and session-frozen `voice_policy_snapshot` bindings, while agent/persona/knowledge control-plane rows mostly only carry `created_by/updated_by` audit fields. Future enterprise work should add org/team compatibility readers and rollout bindings first; directly cloning current asset tables into org-owned rows before membership/authz seams exist will fork the runtime/report truth line.
- M021/S04 quality-event gotcha: `common.ai.llm_service.LLMService.evaluate()` still returns `Result.ok(...)` with hardcoded `60/60/60/60/60` plus generic Chinese coaching copy when the model response cannot be parsed as JSON. If a future report/evaluation surface suddenly looks like a uniformly mediocre success, check for this parse-default path before treating it as a real low-quality evaluation.
- M021 prompt-control-plane gotcha: `PromptTemplateService` lookup alone does **not** prove the model consumed that template. Today `StagedEvaluationService.evaluate_stage()` and `ComprehensiveReportService._generate_detailed_feedback()` both resolve a prompt template first, but `common.ai.llm_service.LLMService.evaluate()` / `generate_report()` rebuild hardcoded prompts internally and ignore the looked-up template text.
