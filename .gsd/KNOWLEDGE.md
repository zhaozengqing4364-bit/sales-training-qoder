- Report-page focused tests need an explicit assertion that `导出报告` is absent; otherwise the dead affordance can drift back into `web/src/app/(user)/practice/[sessionId]/report/page.tsx` without breaking the existing positive-path assertions.
- Standard PPT 这条链的 live contract 在 `/api/v1/presentations`，不是旧的 `/api/v1/admin/presentations`：当前 admin 详情页与用户演练入口都直接消费 `api.presentations`，而 legacy admin backend 仍引用过期 schema；做 replace/version_number/status 修复时应扩展 live surface。
- 本地要跑 S04 presentation/runtime UAT 时，`cd backend && PYTHONPATH=src venv/bin/uvicorn main:app --port 3444` 可能因缺少 `redis` Python 包在启动阶段崩溃并报 `redis package is required for SessionStateService`；先补齐该本地依赖，否则浏览器验证会被环境问题卡住而不是代码回归。
- S04 的 task-level auto verification 不要把 backend/web 检查串成一条 `Verify: ... && cd ../web && ...`；auto-mode 会把它拆成独立检查，导致裸 `pytest ...` 和 `cd ../web` 从 repo root 运行后误报失败。把 backend 与 web 命令写成两条独立校验更稳。
- `backend/tests/integration/test_knowledge_flow.py` 依赖 `/api/v1/auth/dev-login` 夹具；若你改用 repo-root `backend/venv/bin/python -m pytest -c backend/pyproject.toml ...` 运行，可能因为 cwd / env 装载差异让 dev-login 变成 `403`、进而把会话创建打成 `401`。这组知识链回归请继续从 `backend/` 目录下按计划命令运行。
- 本地 `20706b4b-bb22-484a-8f2f-8ecacc43bb3b` 这份 ready standard deck 在 admin 详情页会批量请求缺失 thumbnail 并产生大量 `404 Thumbnail not found` 噪声；它不影响 replace blocker / version/status 合同，但会污染浏览器 console/network 错误列表，排查时别把它误判成 replace 回归。
- S05 改销售语义时，优先保留 `pass_flags.pass_3min_flow/pass_5turn_defense/pass_4step_structure` 和 `SessionReport.logic_score/accuracy_score/completeness_score` 这些既有键名，只在 write layer 里重映射它们的含义；如果直接改 key，report/replay/history/admin 多个消费面会被迫一起改，容易重新引入事实漂移。
- S05 的 persona objection tuning 现在把价格 / 竞品 / ROI 证据类问题并入 `stepfun_knowledge_helpers` 的 widened entity-query path（默认 `top_k=7`、snippet 420、keyword candidate limit 48）；如果旧测试还盯着 5/360，先确认是不是在断言过期基线，而不是把它误报成检索回归。
- 本地做 S05 sales realtime/browser UAT 时，如果 `/practice/{sessionId}` 能连上 backend 但 websocket 一直 `1006` 重连，并且 backend 日志报 `connecting through a SOCKS proxy requires python-socks`，先在 `backend/` 下执行 `venv/bin/python -m pip install python-socks` 再重启 `PYTHONPATH=src venv/bin/uvicorn main:app --port 3444`；否则你会误以为是 ScorePanel 或 report 语义回归，实际是本地 StepFun 上游代理依赖缺失。
- S05 之后如果 `test_apply_latest_scores_to_session_supports_legacy_overall_and_marks_zero_turn_not_evaluable` 还在断言 `90/82/80`，那是过期基线：旧 `score_snapshot.overall + 专业度/沟通技巧/销售流程` 现在会先归一化到 sales rollup，所以 zero-turn legacy fallback 应回落成 `84/84/84` 这类 overall-based sales 分数，而不是继续沿用旧 generic 维度。
- 本地 S05 report/runtime UAT 时，`/practice/{sessionId}/report` 页会并行拉 optional comprehensive-report / highlights 增强接口；这些接口出现 `404/400/500` 或 backend `REPORT_GENERATION_FAILED` 噪声时，先看 unified `/api/v1/practice/sessions/{id}/report` 与页面上的“综合洞察暂不可用 / 高光片段暂不可用”降级文案是否正常，不要把 optional enhancement 失败误判成 sales evidence contract 回归。
- `web/src/app/(user)/practice/[sessionId]/report/page.test.tsx` 里如果要断言“综合洞察暂不可用 / 高光片段暂不可用”文案，必须用 `findByText` / `waitFor` 等待 post-report effect 完成；这些提示是在 unified report 首屏渲染后，随 enhanced-report / highlights 异步请求失败才出现，直接用同步 `getByText` 很容易把真实 UI 降级能力误报成回归。
- 写 projection-backed admin/history 测试时，如果想让 `SessionEvidenceService.build_projection(...)` 真正读取 `score_snapshot.dimension_scores`，就把 `PracticeSession.logic_score/accuracy_score/completeness_score` 设为 `None`；这些列一旦有值，projection 会优先使用 session 行分数而不是消息快照，测试很容易误以为 route 仍在走 legacy SQL。
- 本地用 browser route mock 验证 `/admin/users/{id}` 这类页面时，别直接 mock `http://localhost:3444/api/v1/...` 的 cross-origin `/progress` 请求：Playwright 路由在这个项目里容易把它打成 `ERR_FAILED`/CORS 样噪声，页面只会落到 network error。要验证 inline empty/error state，优先在页面里临时覆写 `window.fetch` 只拦截 progress URL，再点页面上的 `刷新`。
- 本地验证 `/support/runtime` 的 empty/error 分支也别直接 route mock `http://localhost:3444/api/v1/support/runtime/*`：跨 origin mock 同样容易掉进 `ERR_FAILED`/CORS 样噪声。先在已登录页面里临时覆写 `window.fetch` 只拦截 support runtime overview/faults，再点顶部 `刷新`，更容易拿到真实的局部降级 UI。
- S07 本地 browser/report UAT 要让 frontend 和 backend 走同一个 loopback host（`localhost` 对 `localhost`，或 `127.0.0.1` 对 `127.0.0.1`）；如果混用，`/auth/dev-login` 写下的 host-only cookie 不会跟着 shared `/practice/{sessionId}/report` 请求走，页面会假性 401，看起来像 report 回归其实是本地 host 对不齐。
- S07 本地 runtime proof 不要偷懒走 presentation StepFun websocket 的 `type:"text"` shortcut：它能把会话跑到 completed，但当前不会给 user message 落 `transcript_metadata.page_number`，最后 `/practice/{sessionId}/report` 只会得到 `missing_page_metadata` degraded payload。要证明页级 evidence 真落库，继续用真实音频 chunk + `page_change` 路径。
- auto-mode 的最终 verification gate 可能直接从 repo root 执行 `venv/bin/alembic upgrade head` 与 `venv/bin/python -m pytest -c pyproject.toml tests/...`，不会替你 `cd backend`：这个仓库要保持 repo-root `alembic.ini`、repo-root `pyproject.toml` 和 `tests -> backend/tests` shim 可用，同时 `backend/src/common/config.py` 也要能在 repo-root 运行时补读 `backend/.env`，否则 slice 内部验证全绿、最终 gate 仍会因找不到 Alembic / pytest 配置而假性失败。
- `/support/runtime` 的 authoritative release-health line 必须基于 persisted session evidence + runtime diagnostics 做 blocking / warning 分类，而不是只看 `PracticeSession.status`：`status="scoring"`、optional enhancement 噪声、或 degraded PPT/report 会话如果脱离 canonical evidence 语义去判断，很容易被误报成“已完成且健康”或“核心回归”。
- 本地/auto-mode 如果并行跑多个 backend `pytest` 命令（尤其带默认 `pytest-cov` 配置），可能在 coverage combine 阶段互相删掉 `.coverage.*` 临时文件，表现成“所有测试都过了但命令最终 INTERNALERROR / FileNotFoundError 退出”；这类 backend suites 需要顺序跑，别把多个 pytest job 并发起来。
- `web/src/hooks/websocket/message-handlers.ts` 里的 `score_update` 去重不能只看 `overall_score + turn_count`：sales 实时链路会在同一轮内刷新 `dimension_scores`、`stage_name` 和 `suggestions`，如果前端把这些同轮细化更新吞掉，ScorePanel 会静默停留在旧阶段/旧建议。
- `cd web && npm test -- --run '...RightPanelContent.test.tsx'` 这类 Vitest 命令即使把一个不存在的测试文件路径混进去也可能继续返回 0，只报告实际匹配到的文件；跑 S02/S03 这类 slice gate 时要看输出里的 Test Files 列表，别把“命令绿了”误当成缺失的 targeted file 也被执行了。
- sales practice 页的新一轮用户 final transcript 到来时，前端 reducer 必须主动清掉上一轮的 `actionCard` / `fuzzyDetections`；否则即使后端 arbiter 已把同轮 coaching 节流住，旧提示也会在 UI 上跨 turn 残留，看起来像 pacing 仍然失效。
- StepFun sales realtime 的重连快照只该持久化最小 `feedback_pacing_state`（上一条 action 的 signature + turn）；如果 restore 直接重放 `_latest_action_card` 之类读侧诊断字段，会把同一轮动作卡 burst 回放成假回归。
- S03 classic 路由里如果 `action_card` 还停留在 suggestion/pass-flags 文案，先看 `RealtimeFeedbackArbiter.decide()` 调 `build_action_card(...)` 时有没有把 `stage_context` / `score_context` 继续传下去：`CapabilityProcessor` 已经会原样转发 raw `stage_data` 和 `score_payload`，真正容易漏的是 arbiter 的最后一跳；一旦补上，这个改动也会立刻影响现有 StepFun action-card 断言，属于 T03 预期 carry-forward，不是 classic 回归。
- StepFun sales parity要同时保住两条线：对外继续维护稳定的 `_latest_score_snapshot` / `score_update` snapshot 形状，对内另外传一份 raw `score_context_for_arbiter`（保留 `dimensions[*].delta/trend`）和 `_latest_stage_data`（保留 `guidance` / `key_actions` / `progress`）；不要为了 richer arbitration 去改公开 snapshot 结构。
- `SessionEvidenceService` 做 sales read-side alignment 时，不要直接拿“最后一条带 `score_snapshot` 的消息”；要倒序找“最后一条仍然能让 `resolve_sales_report_alignment(...)` 得到 `alignment_used=True` 的消息”。否则后续只落了 `overall_score` 的 partial snapshot 会把 report/replay/history/admin 重新打回 stale `effectiveness_snapshot`。
- `backend/tests/unit/test_replay_service.py` 这类 `MagicMock` session fixture 如果没显式设 `presentation_id = None`，`resolve_scenario_type(...)` 会把 truthy mock 属性误判成 presentation session，进而掉进 `presentation_review` 分支报 `[PRESENTATION_REVIEW_BUILD_FAILED:]`；写 sales replay 单测时把 `presentation_id` 和 `voice_policy_snapshot` 都显式置空更稳。
