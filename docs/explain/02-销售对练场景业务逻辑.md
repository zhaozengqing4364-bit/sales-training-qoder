## 一、场景概述

### 1.1 销售对练是什么？

想象一下，你要训练一批销售新人。传统方式是：
1. 让老销售扮演客户 → 费时间，老销售也很累
2. 让新人直接打电话给真实客户 → 成交机会成本高，客户体验差

**销售对练场景的价值**：让 AI 扮演各种类型的客户，新人可以反复练习销售话术，犯错成本为零。

## 三、销售对练完整流程

### 3.1 步骤一：选择场景

用户打开系统后，看到的场景：

```mermaid
flowchart TB
    A[用户进入系统] --> B[选择：销售对练]
    B --> C{选择 Agent}
    C -->|销售新人培训| D[B2B 软件销售]
    C -->|进阶训练| E[解决方案销售]
    C -->|高级训练| F[价格谈判]
    D --> G{选择 Persona}
    E --> G
    F --> G
    G -->|初级难度| H[友好型客户]
    G -->|中级难度| I[质疑型客户]
    G -->|高级难度| J[强硬型客户]
    H --> K[开始演练]
    I --> K
    J --> K
```

### 3.2 步骤二：创建会话

当用户点击「开始演练」时，系统在后台做的事情：

```mermaid
sequenceDiagram
    participant 用户 as 用户
    participant 前端 as 页面
    participant 后端 as 服务器

    用户->>前端: 点击"开始演练"

    前端->>后端: POST /api/v1/practice/sessions<br/>请求创建会话
    Note right of 后端: 检查用户权限<br/>验证 Agent 存在<br/>创建会话记录
    后端-->>前端: 返回会话 ID

    前端->>后端: WebSocket 连接<br/>/ws/sales/{session_id}
    Note right of 后端: 加载 Agent 配置<br/>加载 Persona 配置<br/>初始化 AI 能力模块
    后端-->>前端: 连接成功，可以开始说话

    前端->>用户: 显示"请说话"
```

### 3.3 步骤三：实时对话交互

这是最核心的部分。当用户说话时，系统会经历多个处理阶段：

```mermaid
flowchart TB
    subgraph "用户说话后的处理流程"
        A[用户对着麦克风说话] --> B[前端采集音频]
        B --> C[发送到服务器]
        C --> D[语音识别 ASR]
        D --> E[得到文字内容]
        E --> F{开始 AI 处理}
    end

    subgraph "AI 能力模块（同时进行）"
        F --> G[模糊话术检测]
        F --> H[销售阶段识别]
        F --> I[知识库检索]
        F --> J[实时评分]
    end

    G --> K[检测到模糊词]
    H --> L[识别当前阶段]
    I --> M[返回产品知识]
    J --> N[计算当前得分]

    K --> O[记录问题]
    L --> O
    M --> O
    N --> O

    O --> P[AI 生成回复]
    P --> Q[语音合成 TTS]
    Q --> R[推送给用户]
```

### 3.4 对话过程中的 AI 能力

在用户和 AI 对话的过程中，AI 的大脑（能力模块）同时在做这些事情：

## 四、销售阶段模型

### 4.1 标准销售流程

AI 会识别用户当前处于哪个销售阶段：

```mermaid
stateDiagram-v2
    [*] --> 开场: 打招呼，破冰
    开场 --> 需求确认: 了解客户需求
    需求确认 --> 方案介绍: 推荐产品方案
    方案介绍 --> 异议处理: 客户提出质疑
    异议处理 --> 促单: 尝试成交
    促单 --> 成交: 客户同意购买
    成交 --> [*]

    异议处理 --> 需求确认: 回到重新了解需求
    促单 --> 异议处理: 客户又有异议
```

### 4.2 每个阶段的 AI 策略

| 阶段 | AI（扮演客户）的目标 | 典型反应 |
|------|-------------------|---------|
| **开场** | 判断是否值得继续 | 冷淡/礼貌/积极 |
| **需求确认** | 透露需求信息 | 给出模糊需求 |
| **方案介绍** | 质疑方案 | 问价格、效果、竞品对比 |
| **异议处理** | 制造更多障碍 | 「太贵了」「不需要」「和 XX 有什么区别」 |
| **促单** | 做出最终决定 | 「我再想想」「和老板商量」 |
| **成交** | 结束对话 | 同意/拒绝 |

## 五、知识库 grounding 机制

### 5.1 什么是知识库？

**知识库 = AI 的「产品手册」**

当用户向客户介绍产品时，AI 需要知道产品的真实信息，这就需要查询知识库：

```mermaid
sequenceDiagram
    participant 用户 as 销售人员
    participant AI as AI 客户
    participant 知识库 as 产品知识库

    用户: 我们这个产品可以帮助企业
    用户: 提升 30% 的销售效率...

    AI->>知识库: 查询"销售效率提升"
    知识库-->>AI: 返回相关产品资料

    AI: 听起来不错，
    AI: 但你们和竞品相比有什么优势？
```

### 5.2 知识库检索流程

```mermaid
flowchart LR
    A[用户说的话] --> B[提取关键词]
    B --> C[向知识库搜索]
    C --> D{找到相关内容？}
    D -->|是| E[返回产品资料]
    D -->|否| F[AI 自由回答]
    E --> G[AI 参考资料生成回复]
    F --> G
```

### 5.3 知识库锁机制（KB Lock）

系统有一个特殊机制：**强制 AI 基于知识库回答**

```mermaid
flowchart TB
    A[用户发言] --> B{是否启用 KB 锁？}
    B -->|否| C[正常 AI 回答]
    B -->|是| D{是否已绑定知识库？}
    D -->|否| E[阻止发言，提示绑定知识库]
    D -->|是| F{知识检索成功？}
    F -->|失败| G[提示错误]
    F -->|无结果| H[提示无足够依据]
    F -->|有结果| I[返回 grounding context]
    I --> C
```

## 十、总结

销售对练场景的核心逻辑：

1. **配置层**：管理员配置 Agent（场景）和 Persona（角色）
2. **会话层**：用户创建会话，AI 加载配置
3. **交互层**：实时语音对话，多个 AI 能力模块同时工作
4. **评估层**：实时评分 + 结束后生成详细报告

这套机制让销售培训变得：
- **低成本**：不用占用老销售的时间
- **可重复**：可以无限次练习
- **标准化**：每个学员面对相同的"客户"
- **可量化**：通过数据了解学员水平
