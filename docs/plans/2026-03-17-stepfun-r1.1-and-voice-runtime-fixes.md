# 阶跃星辰 `step-audio-r1.1` 切换与语音演练修复清单

## 1. 目标

本次调整有两个目标：

1. 将系统默认的 StepFun Realtime 语音模型切换到官方当前最新可用的 `step-audio-r1.1`。
2. 把前序讨论过的语音演练问题、修复方向和后续改造项整理成一份可执行文档，避免后续反复对齐。

本文件区分两类内容：

- `已实施`：本次已经改进到代码里的内容。
- `后续可增强`：本次没有直接编码，但已经明确了方向和落点的内容。

## 2. 官方模型结论

核对日期：2026-03-17

官方文档结论：

- StepFun Realtime API 当前支持 `step-audio-r1.1`。
- `step-audio-r1.1` 是当前实时语音线上的最新主推模型，适合作为新的系统默认模型。

补充说明：

- 官方文档目前存在一处轻微不一致：
  - `llm/realtime` 模型页已列出 `step-audio-r1.1`
  - 旧的 Realtime 参数页仍主要展示 `step-1o-audio`、`step-audio-2`、`step-audio-2-mini`
- 因此本次实现按“官方模型页最新能力”切换到 `step-audio-r1.1`。
- 如果上线验证时上游账号白名单或接口返回模型不支持，应临时回退到 `step-audio-2`，并同步核对账号权限与平台文档更新状态。

参考：

- https://platform.stepfun.com/docs/llm/realtime
- https://platform.stepfun.com/docs/zh/api-reference/realtime/chat
- https://platform.stepfun.com/docs/pricing/details

## 4. 已确认问题与本次实现结论

### 4.1 用户词典 / 同音词归一化

问题：

- 语音转文字时，同音词、近音词会把产品术语识别错。
- 仅依赖模型侧热词不足以稳定解决业务术语问题。

结论：

- 已完成，且已经进入销售与 PPT 双链路。

### 4.2 知识库强制模式误伤训练流程

用户反馈的错误包括：

1. 检索超时。
2. 语音转写未完成就被强制拦截。
3. 未检索到内部依据即直接拒答。

根因：

- 这是应用层 `KB lock` 策略过硬，不是单纯模型能力问题。

### 4.3 AI 单轮提问过多

问题：

- 一次问多个问题，学员接不住。

### 4.4 PPT 演讲评分能力不足

当前问题：

- 现有 PPT 评分偏轻，无法支撑 10 到 30 分钟的完整讲解评分。
- 不应只靠单次 LLM 总结做全局评分。

关键原则：

- 长时段演讲评分必须基于“证据聚合”，不是单轮 prompt。

### 4.5 RAG 提升方向

建议主链路：

- `ASR normalization -> query rewrite -> hybrid retrieval(BM25 + vector + metadata filter) -> rerank -> optional graph expansion -> cited answering`

结论：

- 本次已补齐 `Hybrid + Rerank` 的运行时可控实现。
- GraphRAG / 知识图谱仍不是第一优先级，除非后续问题天然依赖多跳关系推理。

## 5. 后续可增强项

- 真正的 cross-encoder / reranker 模型替换当前轻量级重排逻辑
- 基于知识图谱的多跳关系检索
- 演讲报告接入更强的 LLM 评语精修层
- 基于 `transcript_metadata` 的逐页回放与错词对照 UI

## 6. 验证建议

切换默认模型后，建议做一轮 A/B 对比：

1. 同一份 10 到 20 分钟 PPT 讲稿，对比 `step-audio-2` 与 `step-audio-r1.1`
2. 对比指标：
   - ASR 错词率
   - AI 打断率
   - 问题单轮可承接性
   - 长时段讲解后的总结质量
   - 产品术语命中率

## 7. 风险说明

- 如果数据库里存在人工创建的非默认 voice runtime profile，并且仍显式写死为 `step-audio-2`，本次不会自动替换。
- 如果需要“全量 profile 一键切到新模型”，应额外做一次受控数据迁移，而不是在当前迁移里直接覆盖所有 profile。
