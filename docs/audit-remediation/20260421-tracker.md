## Phase Gate Rules

- Phase 1A/1B backend runtime changes must add failing/characterization tests first.
- Phase 1C/1D frontend UX changes must have co-located Vitest/RTL coverage and no fake UI affordances.
- Phase 1E security changes must prove hostile/tampered inputs are contained without changing external contracts.
- Phase 1F/Growth changes must expose rule versions and explainability; no ML/notification/badge hard-coding.
- Phase 2+ architecture work must use small slices with import graph/contract tests before moving modules.
- Final release gate must attempt every command in the test spec, classify failures as historical/introduced/fixed/deferred, and include rollback notes.
