# PRD：26 项代码审计问题整改与增长迭代计划

## 背景
项目已完成全栈代码审计，确认当前系统具备销售训练、PPT 演练、实时语音对话、报告、回放、排行榜、管理后台和复练推荐基础闭环。审计同时发现 26 个需要分层解决的问题，覆盖质量门禁、安全鉴权、API 一致性、实时训练稳定性、复杂度控制、用户体验真实性和用户粘性增强。

## 目标
1. 恢复可执行、可解释、可复现的前后端质量门禁。
2. 消除明确安全误用点和 API 调用分叉。
3. 保护实时训练、音频留痕、报告和回放的核心可信链路。
4. 将首页、历史页、报告页、训练页串成下一轮复练闭环。
5. 在不改变部署/运维范围的前提下，建立后续可分批执行的工程路线。
6. 将 26 项总路线与首批可执行窄切片分离，避免 roadmap backlog 被误当成首批承诺。

## 非目标
1. 不涉及 Docker、容器化、部署、运维、基础设施。
2. 不在规划阶段修改业务源码。
3. 不一次性大重构 StepFun realtime、API client、practice route、report/replay 页面；首批仅允许为这些区域建立边界图、契约快照和回归测试清单。
4. 不引入新功能型依赖；恢复既有测试运行所需依赖需单独记录原因。
5. 不用增长功能掩盖评分、证据、错误状态不可信的问题。
6. 首批不实现问题 11-26；问题 10 首批只冻结复练数据契约。
7. API envelope 首批不做全量迁移，只覆盖新增和 touched endpoint，并保持前端兼容旧结构。

## 用户故事
1. 作为开发者，我希望后端测试依赖齐全、lint 分层可运行，能在修改核心训练链路前获得可靠回归信号。
2. 作为管理员，我希望权限 helper 不存在“看似校验但实际放行”的风险。
3. 作为用户，我希望首页能明确区分真实空数据和接口失败。
4. 作为训练用户，我希望练习页能同时说明连接、麦克风、音频留痕、会话状态的故障和恢复动作。
5. 作为训练用户，我希望结束训练后报告和回放的音频证据状态可信。
6. 作为复练用户，我希望首页、历史页、报告页、训练页都能告诉我下一次具体练什么。
7. 作为移动端用户，我希望关键入口更容易找到，错误和空态能直接给下一步动作。

## 成功指标
1. 高优紧急项全部有测试或验证命令覆盖，且不引入新的全局质量门禁阻断。
2. backend targeted pytest 不再因 rank_bm25、jwt 缺失在收集阶段失败。
3. RoleChecker always-allow 风险被删除或改为真实鉴权包装，并有防回归检查。
4. 管理端 TTS preview 不再直接拼接 NEXT_PUBLIC_API_URL，而是通过统一 API client。
5. 首页失败模块不再显示为 0 数据默认态。
6. 练习页至少可同时展示连接、麦克风、音频留痕、会话状态四类状态。
7. 训练结束跳转报告前，最后音频留痕分片有完成或失败状态。
8. 首批冻结复练数据契约；完整首页今日复练、历史按问题复练、报告下一轮训练卡在核心体验可信后分阶段完成。
9. 首批执行结束时必须有按阶段验证证据、共享文件冲突处理记录和剩余 backlog 列表。

## 范围分级
高优紧急：1-10，先恢复可信开发与核心体验闭环。
中优待排：11-20，在核心链路稳定后增强复练、个性化和即时反馈。
低优可选：21-26，用于增强长期粘性、移动端和空态引导。


## 首批执行范围
1. Phase 0 基线确认、未提交改动保护、命令矩阵、文件所有权矩阵。
2. 问题 1、2、3、4、5、6、9 的最小可验证实现。
3. 问题 7 只做重构边界和测试清单，不做实际拆分。
4. 问题 8 只做统一 response helper 与 touched endpoint 样例，不做全量 envelope 迁移。
5. 问题 10 只做复练契约设计，不实现完整 UI 增长闭环。

## 数据资格口径
1. score_eligible：completed 且 evaluable，可计入榜单、分数、成就、趋势。
2. retry_eligible：具备 main_issue、next_goal、presentation_review 或 retry_entry，可用于复练建议。
3. explanation_only：证据不足或未完成，只用于解释和引导，不用于正向激励。
